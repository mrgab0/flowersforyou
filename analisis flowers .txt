No es solo un frontend. Es un proyecto Next.js Full-stack con base de datos configurada. Ya cuenta con
  conexión a MongoDB (lib/db.ts) y modelos Mongoose (Product y Post), pero actualmente el frontend muestra
  datos estáticos ficticios.

  Para convertirlo en un CMS funcional, nos falta implementar:

  1. Backend y Persistencia (Server Actions / APIs)
   * Crear funciones del lado del servidor (Server Actions o API Routes en /app/api/) para realizar
     operaciones CRUD (Crear, Leer, Actualizar, Borrar) sobre Post y Product.

  2. Autenticación y Autorización (Seguridad)
   * Implementar Auth.js (NextAuth) o un sistema de sesiones para proteger las rutas del administrador y
     evitar que cualquiera modifique el catálogo.

  3. Panel de Administración (UI del CMS)
   * Diseñar un conjunto de vistas protegidas (por ejemplo, en /app/admin/) con:
     * Formularios para crear/editar productos (con campos para precio, stock, SEO, categoría).
     * Un editor de texto enriquecido (como TipTap o Quill) para escribir artículos (Post).
     * Gestión de carga de archivos (imágenes) usando servicios como Cloudinary o Vercel Blob.

  4. Integración Dinámica del Frontend
   * Reemplazar los demoProducts en app/page.tsx con consultas directas a la base de datos usando dbConnect()
     y Product.find().
   * Crear las páginas de Blog dinámicas para listar y leer los artículos almacenados.

Cuando hablas de "vibe codear" con control, la clave no es solo la tecnología, es el **flujo de trabajo (workflow)** que implementas para aislar los errores antes de que se vuelvan sistémicos.

Aquí tienes una estrategia para proteger tu trabajo con la CLI y organizar tus prioridades para minimizar el caos:

---

## 1. Protección de tu flujo con Gemini CLI

Para no "echar atrás todo lo logrado", trata tu código como un entorno de producción incluso en local:

* **Git como tu red de seguridad:** Antes de lanzar cualquier comando de la CLI que modifique archivos en masa, asegúrate de hacer un `git commit` o al menos un `git stash`. Si algo se rompe, `git checkout .` es tu botón de pánico.
* **Atomic Prompts:** No le pidas a la CLI que reescriba toda la arquitectura. Divide tus prompts en funciones puras y pequeñas. Ejemplo: en lugar de "hazme todo el CRUD", pide "genera solo la función de validación de datos para el esquema de Producto".
* **Validación de tipos (TypeScript es tu mejor amigo):** Si no lo haces ya, usa TypeScript estrictamente (`strict: true`). Esto evitará que los "typos" pasen desapercibidos y te causen esos errores en tiempo de ejecución que parecen "loops infinitos".

---

## 2. Prioridades: ¿Qué es más fácil, rápido y estable?

Para evitar los problemas de conexión infinita y loops, te recomiendo seguir este orden, priorizando la **estabilidad del servidor sobre la complejidad del cliente**:

### A. Prioridad 1: Autenticación (Auth.js) - *La base del control*

Es el paso más importante para el "vibe seguro". Si no tienes esto, cualquier cambio que hagas en el CMS es vulnerable.

* **Por qué:** Configurar Auth.js al principio te da un `middleware.ts`. Este archivo protege todas tus rutas `/admin/*`. Si intentas acceder sin sesión, el servidor rechaza la conexión inmediatamente, evitando loops de renderizado en el cliente.

### B. Prioridad 2: Backend y Persistencia (Server Actions) - *Lo menos propenso a errores*

Olvídate de las API Routes (`/app/api/`) por ahora. Usa **Server Actions**.

* **Por qué:** Las Server Actions viven en el mismo entorno que tus componentes, lo que reduce drásticamente la fricción de "conexión al servidor". No tienes que manejar estados complejos de `fetch` o `axios` en el cliente. Solo invocas la función como una promesa.
* **Seguridad:** Al usar `zod` para validar los argumentos de entrada en tus Server Actions, garantizas que si el cliente envía basura, la acción falla de forma controlada y segura, sin romper la base de datos.

### C. Prioridad 3: Panel de Administración (UI)

* **Tip:** No intentes integrar editores complejos (TipTap) al inicio. Empieza con `textarea` simples. Una vez que el CRUD funcione, el editor de texto enriquecido es solo un componente que sustituye al anterior.

### D. Prioridad 4: Integración Dinámica del Frontend

Esta es la parte más "fácil" una vez que los puntos 1 y 2 están blindados. Solo reemplazas los `mockData` por la llamada directa a tu función de base de datos (`dbConnect`).

---

## Resumen de arquitectura para evitar el caos

> **Regla de oro:** Si usas **Server Actions + Zod (validación) + Middleware de Auth**, eliminas el 90% de los problemas de "conexión infinita" y "typos" en la comunicación cliente-servidor.

### Mi recomendación para tu hoja de ruta:

| Paso | Acción | Razón |
| --- | --- | --- |
| **1** | **Instalar Auth.js y Middleware** | Protege todo antes de construir. |
| **2** | **Definir Schemas con Zod** | Evita errores de tipo en la DB. |
| **3** | **Crear Server Actions (CRUD)** | Comunicación directa, sin API routes extra. |
| **4** | **Conectar UI (Formularios)** | Usa `useFormState` de React para manejar errores de forma limpia. |
